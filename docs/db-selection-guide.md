# DB 선택 가이드

> **TL;DR** — 저장할 게 없으면 **DB 없음**. 단순 저장이면 **Neon**. 로그인 / 파일 업로드 / 사용자별 기록이면 **Supabase**. 의심되면 한 단계 낮은 쪽.

실험 서비스에서 데이터를 어떻게 저장할지 선택하는 기준입니다. 단계는 **3가지**입니다.

원칙: **가능한 한 낮은 단계에서 시작**합니다. 필요해지면 그때 위 단계로 옮깁니다.

> **조직에 공유 Supabase 프로젝트가 이미 있다면**, Supabase 단계를 선택했을 때 별도 프로젝트를 새로 띄우지 않고 기존 프로젝트의 자기 스키마만 추가합니다. 운영 부담이 거의 늘지 않으면서 인증 / 스토리지를 함께 쓸 수 있습니다. 셋업 절차(스키마 격리, 트리거, RLS, 클라이언트 설정)는 **Playground 엔지니어링 가이드의 "Supabase 스키마 격리" 문서**를 참고하세요. 이 문서는 어디에 있는지는 [README의 엔지니어링 상세 가이드 섹션](../README.md#엔지니어링-상세-가이드) 참고.

## 1단계 · DB 없음

**언제 사용하는가**
- 사용자 입력을 저장할 필요가 전혀 없을 때
- 결과를 그 자리에서 보여주고 끝나는 서비스
- 정적 콘텐츠로만 구성된 랜딩 페이지

**예시**
- MBTI 테스트 (결과만 보여주고 저장 안 함)
- 단순 정보 랜딩
- 계산기 / 변환기

**장점**
- 가장 빠르고 안전
- 개인정보 사고 가능성이 매우 낮음
- 운영 비용 거의 없음

## 2단계 · Neon

**언제 사용하는가**
- 단순한 데이터 저장이 필요 (예: 이메일 구독 목록, 설문 응답 수)
- 인증 / 파일 업로드 / 실시간 같은 기능은 필요 없음
- PostgreSQL 표준만 있으면 충분

**예시**
- 사전 신청 폼
- 간단한 응답 모음
- 카운터 / 통계 저장

**특징**
- Serverless PostgreSQL
- 무료 플랜으로 실험 단계 서비스에는 충분
- 사용량이 적을 때 자동 sleep → 비용 효율적

## 3단계 · Supabase

**언제 사용하는가**
- 사용자 로그인이 필요할 때
- 파일 / 이미지 업로드가 필요할 때
- 실시간 기능이 필요할 때
- DB + 인증 + 스토리지를 한 번에 묶고 싶을 때

**예시**
- 사용자별 데이터를 저장하는 도구
- 이미지 업로드가 있는 게시판
- 로그인이 필요한 커뮤니티 실험

**주의**
- 기능이 많은 만큼 설정도 더 신중해야 함
- 행 단위 보안(RLS) 설정 **필수**
- 개인정보 처리 범위가 넓어짐 → privacy 체크리스트를 더 꼼꼼히 본다

## 결정 흐름

```
저장할 데이터가 있는가?
  └ 아니오 → 1단계: DB 없음
  └ 예
      └ 사용자 로그인 / 파일 업로드 / 실시간 기능이 필요한가?
          └ 아니오 → 2단계: Neon
          └ 예    → 3단계: Supabase
```

## 서비스별 추천 예시

자주 만드는 실험 서비스 유형별로 어떤 DB 단계를 권장하는지 정리했습니다.

| 서비스 유형 | DB 권장 | 이유 |
| --- | --- | --- |
| 단순 안내 / 랜딩 | DB 없음 | 사용자 입력을 저장할 일이 없음 |
| 운세 / 성향 테스트 (결과 저장 없음) | DB 없음 | 결과를 그 자리에서만 보여줌 |
| TOPIK 문제 풀이 (문제 데이터만 필요) | Neon | 문제 데이터를 읽기 전용으로 조회 |
| TOPIK 학습 기록 저장 | Neon 또는 Supabase | 사용자별 기록 저장. 로그인이 필요하면 Supabase |
| 게임 랭킹 | Neon | 점수 / 닉네임 정도의 단순 저장이면 충분 |
| AI 상담 기록 저장 | Supabase | 로그인 + 사용자별 기록 + 민감 정보 처리 |
| 커뮤니티 / 게시글 / 댓글 | Supabase | 인증 + RLS 필수 |
| 이미지 / 파일 업로드 | Supabase | 스토리지 기능 필요 |

> 표는 일반적인 권장 예시입니다. 같은 유형이라도 요구사항에 따라 단계가 달라질 수 있으니 위의 [결정 흐름](#결정-흐름)을 우선 적용하세요.

## 단계 변경

처음부터 Supabase로 시작하면 불필요한 복잡도를 떠안게 됩니다.
**낮은 단계 → 높은 단계**로 옮기는 것은 비교적 쉽지만, 반대 방향은 데이터 이전 부담이 큽니다.
의심스러우면 한 단계 낮은 쪽을 선택합니다.
